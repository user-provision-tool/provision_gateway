    - Generated Files: during the deployment, there may be files being generated by the LLM, show these files here -- same as Templates, the filenames should be clickable and thus opened in the editor

- in the project details, if there are new files generated, which should be shown at the top, and highlighted with green
- there should be a deploy button in actions, which trigger the deploy procedure


4. when deploying a service, the system should identifies whether there are missing files for deploying the service
- if yes, the system need to generate those files with the LLM service configured (choose one from the saved LLM configs, only connected ones are visible for choose)
    - just like the saved proxy configs, the saved llm configs should also be listed as card, and only one can be enabled
- the missing file generation should pop up for user to input special requirement, and pop up a window for user to confirm when files are generated
    - in the window, user can click to open the generated files in the editor (same as the one opened in source project page, same hightlights), to make manual modifications
    - there should be a button for re-triggering the generation
    - at the same time, these generated files should be listed in the page `source project` column `Generated Files`

5. on the service page, the services should be automatically grouped by users
- each user have a list of services, gourps of different users should be visually separated in the page (separted with horizental empty line like space)
- each services is a folded card, when click open, the detailed inforamtion can be seen
    - provision url (using the actual provision-nginx ports)
    - passwords
        - the password shold be hidden as dots, but there is a eye button to click to view
        - the password can be reset, after the resetting, the nginx should be reloaded with signal
    - the template / config files being used for deploying the service, can click to view them, but not edit
    - names of containers consist the service, and their status

- a test button, to send a test request to the deployed service via the provision url
- there should be action buttons behind each service
    - up, down
    - rebuild, redploy, delete
    - duplicate (swiftly redeploy for another user)

- there should be action buttons under each user name
    - these actions operate the entire user service group
    - up down delete duplicate

- there should be filters to search about (partial matching supported):
    - user name, service name, container names
    - once matches found, highlight the matched characters and shown the service card (still grouped by users)

- during the service building, the corresponding task is displayed in the service card, once clicked jump to the tasks page, and the building log can be seen from there, for the ongoing task, the log is dynamically growing.

7. you need to re-develop the user mechanism (use a opensource project if you can a simple enough one, if no suitable project found, just develop a simple one)
- there should be a user management systmen, thus
    - the user registration should be supported
    - by default registered user have viewer role -- the registration of normal user has to be approved by the admin, so as to be able to login
    - both the admin and the user can modify their own password

- then the admin can only deploy services for the registered user (not as current, for any users), except some special ones (e.g. `shared`, `public`, `internal`, etc. there should be a configurable list of such `special functional users`) -- thus, on the service deploy popup, there should be a searchable dropdown list for the registered and specia users
    - a user has to be in the activated status, thus one's name can appear in the dropdown list
    - the `special functional users` are for deploying service shared by multile users

- the admin users can have access to the entire webui, the viewers can only see the servces page and only their own services + the allowed special ones
- a user's allowed `special functional users` should be set by admin, in a user management page

- NOTE: if you need to implement a user management system yourself, you need to make a detailed design and plan for it first, you can implement it as a new service if needed (in a separated container like other provision system components) 


8. you should also support two ways of this system interacting with the agents:
- just like the configed LLM, the provision system can be configed with an agent using the openai compatible api, thus, all the LLM based generation will go through this agent
- a mcp server (running as another provision system container), thus external agent can make use of this provision system (not only the core user_provision)
    - in this mode, the provision system will route the file generation requests back to the external agent, and the external agent will provide such generated files back to the provision system 
    - this `external agent` will not be put in the configed LLM / agent, as it is essentially not.
    - the interaction would be like `external agent -[ask for deployement of some service]-> mcp -> provision gateway -[ask for file generation]-> external agent -[generated files]-> provision gateway -[deployed service status]-> external agent`: its a back and forth procedure
