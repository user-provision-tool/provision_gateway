# Updated Design — Provision Gateway

> **Created**: 2026-07-03
> **Purpose**: Records all design changes discovered during implementation.

---

## Change 1: PROVISION_API_URL default port

**Original Design** (design.md):
- `PROVISION_API_URL` default: `http://provision-api:8765`

**Issue Discovered**:
- Inside the Docker network, `provision-api` listens on port **8000** (the container port), not 8765 (the host port).
- The docker-compose.provision.yml maps `8765:8000`, so 8765 is only accessible from the host.

**Resolution**:
- Changed `PROVISION_API_URL` default to `http://provision-api:8000`
- Updated in `app/config.py`
- Updated in `docker-compose.gateway.yml` environment section

**Files Affected**:
- `provision-gateway/app/config.py`
- `docker-compose.gateway.yml`

---

## Change 2: Docker network name

**Original Design** (design.md):
- Network: `provision_default`

**Issue Discovered**:
- The actual Docker network name is `users_provision_default` (auto-generated by docker compose based on the `_users_provision` directory name).

**Resolution**:
- Updated `docker-compose.gateway.yml` to use `users_provision_default` as external network

**Files Affected**:
- `docker-compose.gateway.yml`

---

## Change 3: bcrypt library compatibility

**Original Design** (design.md):
- Auth uses `python-jose (JWT) + passlib (bcrypt)`

**Issue Discovered**:
- bcrypt 5.0.0 is incompatible with passlib 1.7.4 (throws "password cannot be longer than 72 bytes" error)
- Using bcrypt directly (`bcrypt.hashpw` / `bcrypt.checkpw`) instead of passlib wrapper works correctly

**Resolution**:
- Changed `app/services/auth_service.py` to use `bcrypt` directly instead of `passlib.hash.bcrypt`
- passlib is still listed as a dependency but not used for bcrypt

**Files Affected**:
- `provision-gateway/app/services/auth_service.py`
- `provision-gateway/app/routers/users.py` (password change endpoint)

---

## Change 4: Dashboard Dockerfile simplification

**Original Design** (design.md):
- Dashboard Dockerfile: Stage 1 Node 20 → npm ci → npm run build; Stage 2 nginx:alpine

**Issue Discovered**:
- `npm ci` requires a package-lock.json; using `npm install` for initial development

**Resolution**:
- Changed to `npm install` in Dockerfile
- Actual implementation matches design otherwise

**Files Affected**:
- `provision-dashboard/Dockerfile`

## Changes from tasks-20260705.md (2026-07-05)

These conflicts were identified by comparing the task specification against
the original design.md. The task spec takes precedence.

### Conflict 1: LLM Configs — multi-config cards like proxy
**Original**: Single row in `llm_config` table. One active at a time.
**Task spec (4)**: "just like the saved proxy configs, the saved llm configs should also be listed as card, and only one can be enabled"
**Resolution**: Add `is_active` field to `llm_config`, support multiple configs,
list them as cards with toggle (same pattern as proxy configs). Only connected
(reachable) configs can be activated.

### Conflict 2: Source project table columns
**Original**: Name, Files (5 shown + count), Templates, Actions
**Task spec (3)**: Name, Templates (clickable), Generated Files (clickable), Actions.
No .git files displayed. Deploy button in Actions.
**Resolution**: Redesign ServicesPage columns. Filter out .git/ directory. Add
Generated Files column. Add Deploy button per row.

### Conflict 3: Services page (deployed instances) — card-based grouped by user
**Original**: Simple table with User, Service, Label, Status columns
**Task spec (5)**: Grouped by user with visual separation. Collapsible cards per
service with detailed info (URL, password with eye toggle, containers, templates).
Per-service actions (up/down, rebuild, redeploy, delete, duplicate). Per-user
actions. Search/filter. Building task link & log streaming.
**Resolution**: Complete rewrite of UsersPage to card-based grouped layout.

### Conflict 4: User management system
**Original**: Simple `admins` table. Deploy to any username string.
**Task spec (7)**: Full user management — registration, admin approval, viewer
role, special functional users (shared/public/internal), admin deploys only for
registered+approved users, searchable dropdown on deploy form.
**Resolution**: New `end_users` table with registration/approval flow.
User management page for admin. Viewer role with restricted access.

### Conflict 5: Agent interaction — MCP mode
**Original**: LLM config supports local_agent and byok modes.
**Task spec (8)**: Two modes — (a) OpenAI-compatible agent for generation,
(b) MCP server container for external agent back-and-forth interaction.
**Resolution**: Add MCP server as a new provision system container. Add agent
routing logic for file generation requests.

### Conflict 6: Deploy missing-file detection + LLM generation workflow
**Original**: LLM generation is standalone (POST /api/llm/generate). Manual.
**Task spec (4)**: System auto-detects missing files during deploy, offers LLM
generation, allows user input and confirmation, generated files appear in
source project page under "Generated Files" column.
**Resolution**: Pre-deploy validation endpoint. Auto-trigger LLM generation.
Confirmation modal with editor for generated files.

### Conflict 7: Source project — zip upload
**Original**: Upload mode accepts JSON dict of filename→content.
**Task spec (2)**: "upload the repo zip"
**Resolution**: Add zip file upload support. Extract zip into source_projects.

### Conflict 8: File editor with diff highlighting
**Original**: Monaco Editor.
**Task spec (3)**: "changes should be highlighted with git changes shown in
next line style". New generated files highlighted green at top.
**Resolution**: Add diff/git-change highlighting to file editor.
Generated/new files get green highlight.

### Conflict 9: Finer system status
**Original**: System status shows provision-api, provision-nginx, docker host.
**Task spec (1)**: "finer system status is needed, i.e. the status of each
sub services of the provision system"
**Resolution**: Expand system status to show per-container health of all
provision system components (gateway, api, nginx, dashboard, proxy configs).
